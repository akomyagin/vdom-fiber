# TECHNICAL_PLAN — vdom-fiber

Детальный технический план: стек, архитектурные решения и поэтапная разбивка.

## Стек

| Категория | Выбор | Обоснование |
|---|---|---|
| Язык | TypeScript 5.7, `strict: true` | типобезопасность для дерева указателей Fiber |
| Модули | ESM (`"type": "module"`), `moduleResolution: bundler` | современный дефолт, без CJS-легаси |
| Сборка библиотеки | `tsc` (`tsconfig.build.json`, эмит в `dist/`) | достаточно для учебной либы, без бандлера |
| Тесты | Vitest 2 + jsdom | быстрый, нативный ESM/TS; jsdom даёт реальный `document` |
| Покрытие | `@vitest/coverage-v8` | встроенный провайдер, без лишних зависимостей |
| Рантайм | браузер / jsdom | `MessageChannel`, `requestIdleCallback`-подобный тик доступны |

Зависимостей времени выполнения (`dependencies`) **нет** — это библиотека без
внешнего рантайма. Docker Compose не нужен (нет сервисного слоя).

### Решение по JSX

**MVP использует hyperscript-API без JSX.** `createElement(type, props, ...children)`
(алиас `h`) вызывается напрямую. Причины:

- убирает babel/tsc-JSX-конфигурацию из критического пути на ранних этапах;
- делает VNode-дерево явным и легко тестируемым;
- ничего не мешает добавить JSX позже как отдельный слой (см. POST_MVP).

Полноценный JSX (через `jsxImportSource` / babel-плагин, компилирующийся в `h`)
вынесен в [POST_MVP_PLAN.md](./POST_MVP_PLAN.md), потому что это отдельная тема
(тулинг), не связанная с ядром рендер-движка.

## Поэтапная разбивка

### Этап 0 — Скелет (готово)

- `package.json` (библиотека, ESM, Vitest+jsdom, TypeScript), `tsconfig.json`,
  `tsconfig.build.json`, `vitest.config.ts`.
- Модули-заглушки со стабами и `// TODO(Этап N): implement`.
- Один тривиальный smoke-тест (`src/index.test.ts`), проверяющий сборку и jsdom.
- DoD: `npm install && npx tsc --noEmit && npm test` проходят чисто.

### Этап 1 — `createElement` + VNode-дерево

- Реализовать `createElement(type, props, ...children)`:
  - нормализация детей: развернуть вложенные массивы, отбросить
    `null`/`undefined`/`boolean`, обернуть `string`/`number` в `TEXT_ELEMENT`;
  - гарантировать, что `props.children` — всегда массив `VNode[]`.
- `createTextElement(value)` для примитивов.
- Тесты: форма дерева для вложенных детей, текстовые узлы, пустые/фильтруемые дети,
  сохранение прочих props.
- DoD: любой валидный вызов возвращает предсказуемое VNode-дерево; ветки нормализации покрыты.

### Этап 2 — Синхронный mount (без диффинга)

- `render(vnode, container)`: рекурсивно создать реальные DOM-узлы
  (`document.createElement` / `createTextNode`), применить props
  (атрибуты, `on*`-обработчики, `style`), обойти детей в глубину, добавить в контейнер.
- Разделить «применение props» в отдельную внутреннюю функцию — переиспользуется в Этапе 3.
- Тесты (jsdom): собранное дерево `container.innerHTML` / обход узлов, навешенные
  обработчики срабатывают, атрибуты выставлены.
- DoD: произвольное VNode-дерево корректно отрисовывается в контейнер за один синхронный проход.

### Этап 3 — VDOM-диффинг (reconciliation), синхронный

- `reconcile(container, prev, next)`: диффинг одного узла:
  - оба null → no-op; `prev` null → mount; `next` null → remove;
  - разный `type` → replace целиком;
  - одинаковый `type` → пропатчить props (добавить/обновить/удалить атрибуты и
    обработчики) и рекурсивно сверить детей.
- Дети: базовое сопоставление по индексу + опциональные `key` для списков.
- Тесты (jsdom): смена текста, добавление/удаление атрибута и обработчика,
  добавление/удаление/переупорядочивание детей, замена узла при смене типа,
  сохранение DOM-идентичности неизменённых узлов (не пересоздаются).
- DoD: повторный `reconcile` применяет минимальные мутации, не перестраивая всё дерево.

### Этап 4 — Fiber-структура данных

- Ввести `Fiber` (`type`, `props`, `dom`, `parent`/`child`/`sibling`, `alternate`,
  effect tag: PLACEMENT / UPDATE / DELETION).
- `performUnitOfWork(fiber)`: обработать один fiber и вернуть следующий
  (child → sibling → подъём по `return`/parent), **без рекурсии**.
- Построение WIP-дерева fiber'ов из VNode-дерева; фаза commit применяет накопленные
  эффекты к реальному DOM за один проход после завершения обхода.
- Тесты: пошаговый обход даёт fiber'ы в ожидаемом DFS-порядке; после ручного
  «прокручивания» всех единиц работы commit даёт то же DOM-дерево, что и Этап 2/3.
- DoD: обход можно остановить после любого fiber'а и продолжить с сохранённого указателя
  без потери состояния.

### Этап 5 — Планировщик (прерывание + приоритеты) — самый сложный этап

- `scheduler.ts`: work loop, который вызывает `performUnitOfWork` в цикле, пока не
  исчерпан временной бюджет кадра (`shouldYield()` по дедлайну), затем уступает
  управление браузеру через `MessageChannel`-тик (порт-postMessage; фолбэк —
  `setTimeout(0)` / `requestIdleCallback`, если доступен) и возобновляет работу.
- Очередь с приоритетами (`Priority`: Immediate/UserBlocking/Normal/Idle);
  высокоприоритетное обновление **вытесняет** уже идущий низкоприоритетный рендер:
  текущее WIP-дерево отбрасывается/перезапускается с новым корнем.
- Разнести на фазы: `render` (прерываемая, строит WIP) и `commit`
  (синхронная, неделимая — применяет эффекты атомарно).
- Тесты: детерминированный контроль «дедлайна» (инъекция таймера/`now()`),
  проверка что длинный рендер разбивается на несколько тиков; высокоприоритетное
  обновление применяется раньше фонового; итоговый DOM корректен после прерываний.
- DoD: длинный рендер не блокирует поток одним синхронным проходом; приоритеты соблюдаются;
  результат идентичен синхронной версии.

### Этап 6 — Hooks (`useState`)

- Пер-fiber список хуков, сопоставление по порядку вызова (rules of hooks).
- `useState(initial)` → `[state, setState]`; `setState` кладёт обновление в очередь
  хука и планирует re-render владельца через scheduler (Этап 5).
- Поддержать функциональные компоненты как `type` VNode (функция → вызов, дети из результата).
- Демо: компонент-счётчик, обновляющийся по клику через свой рендер-конвейер.
- Тесты (jsdom): начальный рендер, `setState` обновляет DOM, несколько независимых
  состояний в одном компоненте, функциональный апдейтер `setState(prev => …)`.
- DoD: счётчик работает end-to-end через собственный VDOM + Fiber + scheduler + hooks.

## Тестовая стратегия

- Основной ярус — интеграционный по духу: рендерим в jsdom-контейнер и утверждаем
  на **итоговом DOM-дереве** после mount/diff/commit, а не на внутренних вызовах.
- Юнит-тесты для чистой логики (нормализация детей в `createElement`, DFS-порядок в
  `performUnitOfWork`, порядок в очереди приоритетов).
- Планировщик тестируется детерминированно: время/дедлайн инъектируются, а не берутся
  из реальных таймеров, чтобы прерывания были воспроизводимы.

## Definition of Done (общий для этапа)

`npx tsc --noEmit` чисто · `npm test` зелёный · новый код покрыт тестами на DOM-результат ·
публичный API отражён в `src/index.ts`.
